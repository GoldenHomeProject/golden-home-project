## Failure 1: Watchdog counts Pi-local network blips as upstream service failures

**Evidence:**
- `{"t":"2026-05-17T07:12:27-04:00","label":"goldenhomeproject.com",...,"http":"000000","status":"fail"}` (checks.jsonl)
- `{"t":"2026-05-17T07:12:27-04:00","label":"meta_graph_api",...,"http":"000000","status":"fail"}` (same second)
- `{"t":"2026-05-17T07:12:27-04:00","label":"ntfy_server",...,"http":"000000","status":"fail"}` (same second)

The same triple-simultaneous-fail pattern recurs at `07:13:29`, `07:14:29`, `11:51:52/11:54:03`, plus 40+ isolated `http:"000000"` (curl connect-fail sentinel, not 5xx) entries across the week. Three independent providers don't fail in lockstep — that's the Pi's local link.

**Root cause:** `ghp-watchdog.sh` treats curl connection failure (`000`) as an upstream outage, so any Pi-side DNS/Wi-Fi glitch burns through `FAIL_THRESHOLD=2` and fires false ntfy alerts.

**Patch:** `pi/watchdog/ghp-watchdog.sh:97`

```diff
--- a/pi/watchdog/ghp-watchdog.sh
+++ b/pi/watchdog/ghp-watchdog.sh
@@ -94,6 +94,16 @@ while true; do
   ts=$(date -Iseconds)
   if [ $((cycle % ROTATE_EVERY)) -eq 0 ]; then rotate_if_needed; fi
   cycle=$((cycle + 1))
+  # Skip the cycle if the Pi itself can't reach the open internet — otherwise
+  # one bad Wi-Fi minute lights up all 3 targets and burns the fail threshold.
+  # 1.1.1.1 is a fixed-IP anycast endpoint (no DNS needed, no token, free).
+  if ! curl -sf -o /dev/null -m 5 https://1.1.1.1/ 2>/dev/null; then
+    for entry in "${TARGETS[@]}"; do
+      IFS='|' read -r label url regex <<< "$entry"
+      printf '{"t":"%s","label":"%s","url":"%s","http":"000","status":"skip"}\n' \
+        "$ts" "$label" "$url" >> "$LOG_FILE"
+    done
+    sleep "$INTERVAL"; continue
+  fi
   for entry in "${TARGETS[@]}"; do
     IFS='|' read -r label url regex <<< "$entry"
     result=$(check_one "$label" "$url" "$regex")
```

**Why safe:** Pure gate added before existing per-target logic — when the network is up (the >99% case), behavior is byte-identical; when it's down, the cycle logs `status:"skip"` rows that don't increment fail counters and never trigger ntfy. No new service, no token, no dependency change.

**Verification:** `sudo ip link set wlan0 down`, wait ~3 cycles, confirm `tail ~/ghp-watchdog/logs/checks.jsonl` shows `"status":"skip"` (not `"fail"`) and that no ntfy push fires; `ip link set wlan0 up`, confirm next cycle logs `"status":"ok"`.
